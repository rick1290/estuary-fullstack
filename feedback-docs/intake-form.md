🧪 UX Test Report — Intake Form Builder
Path: /dashboard/practitioner/intake Tester: Claude (acting as new practitioner user) Date: April 26, 2026

✅ What Worked Well
1. Clear entry-point cards on empty state The two-card layout ("Create Intake Form" / "Create Consent Form") with icons and short descriptions makes the starting choice obvious. The visual differentiation is clean. Path: /dashboard/practitioner/intake → IntakeFormCards component
2. "Creating..." loading state on the Create button When submitting the Template Details modal, the button text changes to "Creating..." giving the user clear feedback that something is happening. Path: /dashboard/practitioner/intake → TemplateDetailsModal → button[ref_35]
3. "Question added" success toast After saving a question, a green checkmark toast appears at the top of the page — fast, unobtrusive, and clear confirmation. Path: /dashboard/practitioner/intake → QuestionForm → onSave toast
4. Multiple Choice options render as pills in the question list After saving a Multiple Choice question, the options render as readable tag chips below the question, not just a comma string. Great at-a-glance readability. Path: /dashboard/practitioner/intake → QuestionListItem → options display
5. Question type selector icons Each type button (Short Text, Long Text, Single Choice, etc.) has a small icon — helpful for visual scanning at a glance. Path: /dashboard/practitioner/intake → QuestionForm → TypeSelector
6. Inline editing expands within the list The form editor expands inline below the card rather than navigating away, keeping the user in context. Good pattern for this level of complexity. Path: /dashboard/practitioner/intake → IntakeFormCard → expanded edit panel

⚠️ What Could Be Better
7. No auto-navigation into builder after creating a form After clicking "Create" in the Template Details modal, the user is returned to the list page and must manually click the card to open the editor. This extra step is friction — users expect to land directly in the builder. Path: /dashboard/practitioner/intake → TemplateDetailsModal → onCreate() → should redirect to editor
8. Typing in the modal text fields doesn't work without programmatic input When a user types naturally into the Title or Description fields in the Template Details modal, keystrokes don't register visually (the fields remain empty). This was reproducible — had to use programmatic form_input to populate values. Likely a React controlled-input event binding issue. Path: /dashboard/practitioner/intake → TemplateDetailsModal → textbox[ref_32] and textbox[ref_33]
9. Create button shows no validation error when title is empty Clicking "Create" without a title silently does nothing — the button appears muted but there's no inline error message (e.g. "Title is required") to tell the user why it isn't working. Path: /dashboard/practitioner/intake → TemplateDetailsModal → button[ref_35] → missing validation feedback
10. "Change" link in Template Details modal does nothing visible The "Change" button next to "Intake Questionnaire" doesn't open any type-switcher or give any response. It may be unimplemented or broken. Path: /dashboard/practitioner/intake → TemplateDetailsModal → button[ref_31] "Change"
11. Question count on card shows "questions" with no number The form card displays the word "questions" but no count (e.g. it should say "2 questions"). After adding questions this never updates to show a number. Path: /dashboard/practitioner/intake → IntakeFormCard → question count display (below Active badge)
12. Comma-separated options entry is not user-friendly For Single Choice and Multiple Choice questions, options are entered as a single comma-separated string ("Option A, Option B, Option C"). This is confusing for non-technical users who may not know what "comma-separated" means, and makes it hard to include commas within option text. Should be replaced with an add/remove individual chip input. Path: /dashboard/practitioner/intake → QuestionForm → textbox[ref_132] "Options (comma-separated)"
13. Edit mode is missing the Type selector When editing an existing question, the edit form only shows Question Label, Help Text, and Required — the Type cannot be changed. Users who want to change "Short Text" to "Long Text" must delete and recreate the question. Path: /dashboard/practitioner/intake → QuestionEditForm (inline edit) → missing Type selector
14. Inconsistent Help Text placeholder copy between add and edit modes In the Add Question form, the help text field placeholder reads "Additional context for the client". In the Edit Question form it reads "Optional". Should be consistent. Path: /dashboard/practitioner/intake → QuestionForm (add) vs QuestionEditForm (edit) → help text placeholder
15. Add Question button requires ref-based click (coordinate clicks unreliable) Clicking the "Add Question" button by coordinate sometimes fails silently — the form doesn't open. The button's hit target or z-index may be too small/interfered with. Reliable only when using the element reference directly. Path: /dashboard/practitioner/intake → button "Add Question" [ref_103/ref_146]
16. "Link to Service" modal Cancel button unresponsive to coordinate clicks The Cancel button and X close button in the "Link to Service" modal do not respond to coordinate-based clicks and cannot be found in the accessibility tree, suggesting the modal may not be properly accessible or its close handlers are misconfigured. Path: /dashboard/practitioner/intake → AttachToServiceModal → Cancel button and X close button
17. Browse Templates empty state is too bare "No platform templates available yet." with just an icon gives no next-step guidance. Should include a CTA like "Build your own from scratch" or a note about when templates will be available. Path: /dashboard/practitioner/intake → BrowseTemplatesModal → empty state

🐛 Bugs
18. 🔴 BUG — Typing in modal inputs is non-functional Native keyboard input into the Title and Description fields in the Template Details modal produces no visible output. The fields don't update when typed into. The form_input programmatic method works, confirming it's a React synthetic event issue (likely missing onChange handler or controlled component mismatch). Path: /dashboard/practitioner/intake → TemplateDetailsModal → <input> and <textarea> elements
19. 🔴 BUG — Support widget overlaps the question form and is hard to dismiss When the floating support chat widget ("How can we help?") is open and the user opens the New Question form (which appears at the bottom of the page), the widget overlays the form fields. The X button does not close the widget — it requires pressing Escape (which only fades it) and clicking outside. This blocks access to the question type buttons (Yes/No, Scale 1-10 are obscured). Path: /dashboard/practitioner/intake → SupportChatWidget (bottom-right FAB) z-index conflict with NewQuestionForm
20. 🟡 BUG — Delete question requires two clicks but gives no visual confirmation prompt The first click on the X/delete button turns it red, the second click deletes. This two-click "confirm" pattern is undiscoverable and has no label/tooltip. Most users will think the first click failed rather than that it's awaiting confirmation. Should be replaced with an explicit confirmation dialog ("Are you sure you want to delete this question?"). Path: /dashboard/practitioner/intake → QuestionListItem → Delete question button[ref_102/ref_121/ref_145]
21. 🟡 BUG — "Attach to service" Link button visibility is inconsistent In the Link to Service modal, some services show the "Link" button visibly highlighted while others show it in a greyed/muted state with no apparent logic explaining the difference (not based on already-linked status from what's visible). Path: /dashboard/practitioner/intake → AttachToServiceModal → ServiceListItem → Link button states
